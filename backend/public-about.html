About this page

How the Art Marketplace is structured and what can be improved.

How the application works

Users browse artwork on the marketplace page. The frontend fetches artwork data, allows filtering, and (for authenticated users) supports creating, editing, and deleting artwork entries. Purchases and top spender statistics are handled through dedicated backend endpoints.

Technologies used

  • Frontend: Next.js (App Router), React, TypeScript, Tailwind CSS.
  • Backend: Node.js with Express for REST API endpoints.
  • Database: MongoDB with Mongoose models and schemas.
  • Data exchange: JSON over HTTP using the Fetch API.

Main limitations in this implementation

  • Authentication uses browser storage without robust token/session hardening.
  • The API base URL is currently fixed, making deployment configuration less flexible.
  • Authorization is limited; sensitive operations should be protected server-side per role.
  • Validation and error handling can be expanded for edge cases and better user feedback.
  • No automated end-to-end tests are currently enforcing full user journeys.

Alternative approaches

  • Use Next.js API routes or a monorepo BFF instead of a separate Express server.
  • Use PostgreSQL with Prisma for relational data constraints and migrations.
  • Use JWT + refresh tokens or session-based auth with secure cookies.
  • Use GraphQL for flexible client queries where shape variability is high.
  • Add infrastructure like Redis caching and background jobs for scale-oriented workloads.
